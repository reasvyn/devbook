# Becoming a Professional

## Description

You can build software. You know Git, databases, APIs, and the web. Now you need to bridge the gap between writing code for yourself and working as a professional software engineer. This document covers testing, development workflow, portfolio building, interview preparation, job searching, and what to expect in your first 90 days on the job. By the end, you will be ready to launch your career.

### Software Testing

Testing is what separates hobby projects from professional software. Without tests, every change is risky. You fix one thing, something else breaks, and you do not find out until a user reports it. With tests, you can change code with confidence.

**Why testing matters.** A test suite is a safety net. It tells you, in seconds, whether your changes broke existing functionality. This lets you refactor aggressively, add features without fear, and deploy frequently. In a professional setting, untested code is often treated as broken by default. Teams enforce testing because they have learned the hard way that untested code accumulates bugs faster than any manual testing can catch. Consider a payment system: a single untested edge case — a decimal rounding error, a null response from a payment gateway — can cost real money and real customer trust. Testing is not overhead. It is insurance.

**Types of tests.** There are three main categories you need to know.

A unit test verifies a single function, method, or component in isolation. It replaces external dependencies (databases, APIs, file systems) with fake versions called mocks or stubs. Unit tests are fast — they run in milliseconds — and they tell you exactly what broke and where. You should write unit tests for every piece of business logic you write. For example, if you have a function that calculates shipping costs based on weight, distance, and membership tier, a unit test will pass in various combinations and assert the correct cost comes out. No database needed, no HTTP request — just pure logic verification.

An integration test verifies that two or more components work together correctly. It tests the boundary between your code and a real database, a real API, or the file system. Integration tests are slower than unit tests but catch a class of bugs that unit tests miss — wiring errors, configuration problems, unexpected data formats. A common integration test scenario: insert a record into the database using your repository layer, then query it back and verify the fields match. This catches issues like mismatched column types, missing migrations, or incorrect ORM mappings.

An end-to-end test simulates a real user interacting with your application. It opens a browser, clicks buttons, fills forms, and checks that the right things happen. E2E tests are the slowest and most fragile, but they give you the highest confidence that the system works as a whole. Most teams run a small number of critical E2E tests and rely on unit and integration tests for coverage. An example E2E test for an e-commerce site: navigate to the product page, add an item to the cart, proceed to checkout, enter shipping details, submit the order, and verify the confirmation page displays the correct total.

These three categories form the testing pyramid. Unit tests form the wide base — you write many of them. Integration tests form the middle layer — fewer than unit tests, but still substantial. End-to-end tests form the narrow top — a small number covering the most critical user journeys. Teams that invert this pyramid (many E2E tests, few unit tests) end up with slow, flaky test suites that no one trusts. A practical ratio to aim for: roughly 70% unit tests, 20% integration tests, 10% end-to-end tests.

**Test-driven development.** TDD flips the usual order. Instead of writing code and then testing it, you write the test first, watch it fail, then write the minimum code to make it pass. The cycle is red-green-refactor: write a failing test (red), make it pass (green), then clean up the code (refactor). TDD forces you to think about what your code should do before you write it. It also guarantees that every piece of code has a test — because the test came first. Many professional teams practice TDD for all business logic. It feels slow at first, but you gain speed as you stop debugging regression bugs. A concrete example: suppose you need to implement a function `discount(price, coupon)` that applies a percentage discount. You write the test first: `assert discount(100, "SAVE10") == 90`. It fails because the function does not exist yet. Then you implement the minimum: parse the coupon code, apply 10% off, return the result. The test passes. Then you add another test: `assert discount(100, "INVALID") == 100` — an invalid coupon should return the original price. You implement the fallback. The tests now cover both paths before you have written any production code that is not tested.

**Code coverage.** Coverage is a metric that tells you what percentage of your code is executed by tests. It is measured by tools that instrument your code and report which lines ran during the test suite. Aim for coverage, but do not fetishize it. 100% coverage does not mean bug-free code — you can have tests that run every line but never check the correct behavior. Coverage tells you what was executed, not what was verified. Use it as a signal for untested areas, not as a goal. A healthy target for a professional project is 80-90% coverage. The remaining 10-20% is usually boilerplate, configuration, or error handlers that are tested implicitly through integration tests.

**Writing testable code.** Not all code is easy to test. Code that is tightly coupled, full of global state, or does too many things at once is hard to test. Writing testable code means following principles like dependency injection (pass dependencies in rather than creating them inside a function), single responsibility (each function or class does one thing), and avoiding side effects where possible. When you design for testability, you naturally produce cleaner, more modular code. A good rule of thumb: if a function is hard to test, the design is probably wrong. Compare these two approaches:

```python
# Hard to test — creates its own database connection internally
def get_user(user_id):
    db = Database("production")
    return db.query("SELECT * FROM users WHERE id = ?", user_id)

# Easy to test — database connection is injected
def get_user(user_id, db):
    return db.query("SELECT * FROM users WHERE id = ?", user_id)
```

The second version lets you pass a mock database in tests and a real database in production. The function itself does not need to know which one it is using.

**Testing tools and frameworks.** Each language has its own testing ecosystem. In Python, pytest is the standard — it is simpler and more powerful than the built-in unittest module. In JavaScript, Jest and Vitest dominate for frontend and Node.js testing. For end-to-end testing, Cypress and Playwright are the most popular choices. In Java, JUnit is universal. The specific tool matters less than understanding the concepts. Learn one testing framework well in your primary language, and you will be able to pick up others quickly. The patterns — setup, execute, assert, teardown — are the same everywhere.

**Testing in practice: a real example.** Imagine you are building a REST API endpoint that creates a new user account. Here is how you would test it at each level. A unit test validates the password hashing utility: `assert hash_password("hello123") != "hello123"` and `assert verify_password("hello123", hashed)`. An integration test fires up the application, sends a POST request to `/api/users` with a JSON body, and asserts the response status is 201 and the response body contains the user ID. An end-to-end test opens the registration page in a browser, fills out the form, clicks submit, and confirms the user sees a welcome message. Each level catches different failure modes: the unit test catches a broken hash algorithm, the integration test catches a misconfigured route or missing database column, the E2E test catches a JavaScript error in the frontend form handler.

**Testing legacy code.** Not every codebase you encounter will have tests. Many existing projects have little or no test coverage. When you join a team that works on legacy code, your first testing task is not to cover everything — it is to add tests for the code you are about to change. Before modifying a function, write tests that characterize its current behavior (this is called characterization testing). Then make your changes. If the tests pass, you know you have not broken existing behavior. Over time, this approach incrementally builds a safety net around the most volatile parts of the codebase. A team that adds tests alongside every bug fix will, after a year, have a well-tested codebase without a dedicated "testing phase."

**Common testing mistakes to avoid.** Testing implementation details instead of behavior. A test that checks that a private method was called is brittle — rename that method and the test breaks even though the behavior is unchanged. Instead, test the public interface and assert on the observable result. Another mistake is testing the framework. Do not write tests that verify your framework works as advertised (assuming the framework has its own tests). Focus on testing your logic, not the framework's. A third mistake is flaky tests — tests that sometimes pass and sometimes fail due to timing, ordering, or environmental issues. A flaky test is worse than no test because it erodes trust in the entire suite. If a test is flaky, fix it or remove it immediately.

For a deep dive into testing concepts, see the [Software Testing](../../software/software-testing/index.md) subject in DevBook.

### Development Workflow

Professional software development follows a structured process. It is not about typing faster — it is about coordinating with a team, maintaining quality, and shipping reliably.

**Feature workflow.** The typical flow for a new feature looks like this. A ticket or user story describes what needs to be built. You create a new branch from the main branch. You write code, including tests. When you are done, you open a pull request. Your teammates review the code. They leave comments, ask questions, and request changes. Once everyone approves, the code is merged. Then it goes through a deployment pipeline — automated tests run, the code is built, and it is deployed to production. Every step exists for a reason. Skipping steps creates risk. For example, merging directly to main without a pull request bypasses code review, which means bugs, style inconsistencies, and design issues are caught after they reach production. Each step in the workflow is a gate that catches problems before they reach users.

**Writing a good pull request.** The quality of your pull request determines how easy it is for your teammates to review and merge it. Start with a clear title that summarizes the change: "Add payment retry logic for failed Stripe charges" is better than "Fix stuff." Write a description that explains what the change does and why. Include screenshots or screen recordings for UI changes. Reference the ticket number so the reviewer can see the original requirement. Keep the PR focused on one concern — if you notice a formatting issue in an unrelated file, file a separate ticket instead of fixing it in the same PR. A focused PR is reviewed faster and merged sooner. If your PR touches many files, consider splitting it into multiple smaller PRs that build on each other. Each PR should be independently deployable — if merging it breaks production, you should be able to revert it without reverting unrelated changes.

**Branching strategies.** Teams use naming conventions to keep branches organized. A common pattern is to prefix branches by type: `feature/` for new features, `fix/` for bug fixes, `chore/` for maintenance tasks. For example, `feature/user-authentication` or `fix/login-redirect`. The main branch (often called `main` or `master`) is protected — you cannot push directly to it. All changes go through pull requests. Some teams use Git Flow with separate `develop` and `main` branches, but most modern teams use trunk-based development where everyone merges to `main` frequently and branches live only a few days. The key rule in any strategy: keep branches short-lived. A branch that lives for weeks will drift far from main and cause painful merge conflicts. Aim to merge a branch within one or two days of creating it. If a feature is too large for that, break it into smaller pieces that can be merged incrementally behind feature flags.

**Feature flags.** Feature flags (also called feature toggles) let you merge incomplete code to main without releasing it to users. The code checks a configuration flag at runtime: if the flag is on, the new feature is visible; if off, the old behavior runs. This enables trunk-based development for large features because you can merge small, tested increments without waiting for the entire feature to be complete. Once the feature is fully built and tested, you flip the flag on for all users. Later, you remove the flag and the old code path. Feature flags also enable canary releases — roll out a feature to 1% of users, monitor for errors, then gradually increase. As a junior, you will likely not design feature flag systems, but you should understand the concept and use flags when your team adopts them.

**Code review.** Code review is not about catching mistakes, though it does that too. It is about knowledge sharing, maintaining consistency, and building collective ownership of the codebase. When you review someone else's code, be specific, be kind, and focus on the code — not the person. Say "this variable name is unclear" rather than "you named this wrong." When you receive a review, do not take it personally. Your code is not your identity. Every comment is an opportunity to learn. If you disagree with a suggestion, explain your reasoning. The goal is the best outcome for the code, not for anyone to be right. Keep pull requests small — a PR that touches 20 files across 5 different concerns is hard to review thoroughly. Aim for focused, incremental changes.

**What to look for in a code review.** As a junior, you might wonder what you can possibly contribute to a code review. More than you think. Readability: can you understand what the code does? If not, suggest clearer naming or comments. Test coverage: do the new tests actually test the right thing? Are edge cases covered? Bugs: look for off-by-one errors, null pointer risks, incorrect conditionals. Consistency: does the code follow the project's existing patterns for error handling, logging, and formatting. Security: are user inputs validated? Is there an SQL injection risk? Is sensitive data exposed in logs? Start by leaving a few thoughtful comments on each PR you review. You will get better with practice, and your teammates will appreciate the extra eyes.

**Receiving code review feedback gracefully.** When a reviewer points out an issue, your first reaction might be defensive. Pause. Thank them for catching it. If you do not understand the feedback, ask clarifying questions. If you disagree, explain your reasoning with specific examples. "I used a switch statement here because the logic has three distinct cases and the switch makes the control flow visible at a glance" is a good response. "I prefer it this way" is not. Remember that the reviewer is investing their time to make the code better — they are on your side.

**CI/CD.** Continuous Integration means that every time you push code, automated tests run. If the tests fail, the team knows immediately. Continuous Deployment (or Delivery) means that passing code is automatically deployed to a staging or production environment. CI/CD pipelines are configured using tools like GitHub Actions, GitLab CI, or Jenkins. You do not need to master these as a junior, but you should understand the concept and be able to read a basic pipeline configuration. A simple pipeline has stages: install dependencies, run linters, run tests, build the application, deploy. Each stage must pass before the next begins. If any stage fails, the pipeline stops and notifies the team. Green builds (all tests passing) are the default expectation. A broken build is a team emergency — fix it before starting anything new.

**A sample CI pipeline walkthrough.** Here is a typical GitHub Actions configuration for a Node.js project:

```yaml
name: CI
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: 18
      - run: npm ci
      - run: npm run lint
      - run: npm test
      - run: npm run build
```

Every time anyone pushes code, this pipeline runs four steps: install dependencies exactly from the lockfile, run the linter to check formatting, run the test suite, and verify the project builds successfully. If any step fails, the push is marked red and the team is notified. Understanding this file — what each line does and why each step exists — will put you ahead of most juniors.

**Remote work and asynchronous communication.** Many engineering teams work fully remotely or in a hybrid model. This changes how collaboration happens. You write more — documentation, written updates, clear pull request descriptions — because you cannot tap someone on the shoulder. Over-communicate in writing. Assume good intent when reading messages. Be responsive in team chat during working hours. Use status indicators to show your availability. When you are stuck, post a clear description of the problem in the team channel rather than waiting for a meeting. Remote work requires discipline and intentional communication, but it also offers flexibility and focus time. Learn these skills early — they will serve you throughout your career regardless of where you work.

**Best practices for written communication.** When you ask for help in a team chat, structure your message like a bug report: what you are trying to do, what you tried, what happened, and what you expected. This format makes it easy for someone to help you quickly without a back-and-forth. For longer discussions, prefer a document or forum thread over a chat channel — chat buries information in scrollback. Use threaded replies to keep conversations organized. Set your status when you are in deep focus mode so teammates know you might respond slowly. A simple "Focusing until 3 PM" status prevents interruptions while setting clear expectations. When you receive a question from a teammate, even if you do not know the answer, acknowledge it: "I saw your question and am looking into it" is better than silence.

**Agile and Scrum.** Most professional teams use some form of agile methodology. Scrum is the most common framework. Work is organized into sprints — fixed-length periods (usually one or two weeks) during which the team commits to completing a set of work. Each sprint starts with a planning meeting where the team decides what to work on and estimates effort. Each day starts with a standup — a brief meeting where each person says what they did yesterday, what they will do today, and any blockers they have. Standups are not status reports for managers — they are coordination meetings for the team. At the end of the sprint, the team holds a review (demonstrate what was built to stakeholders) and a retrospective (discuss what went well, what could improve, and what to change starting next sprint). These ceremonies create rhythm, accountability, and continuous improvement. Not every team follows Scrum strictly — many adopt hybrid approaches. The key idea is iteration: build something small, get feedback, improve, repeat.

**How to make standups productive.** Many juniors treat standups as a stressful performance review. Instead, think of them as a coordination tool. Prepare your update before the meeting: bullet points of what you completed, what you are working on today, and any blockers. If you are stuck on something, say so clearly: "I am blocked on the user authentication PR because I am unsure how to handle token refresh. I would like to pair with someone on it after standup." This gives the team a concrete action item. If you have nothing to report because you were stuck all day, say that too — hiding a blocker only delays help. Keep your update under 60 seconds. If a topic needs deeper discussion, take it offline after standup rather than derailing the meeting.

**Sprint retrospectives — what to say.** The retrospective is your chance to shape how the team works. When things went well, call it out: "I appreciated that we kept PRs small this sprint — reviews were much faster." When things went poorly, propose a specific improvement instead of just complaining: "We had three production incidents this sprint because database migrations were not reviewed. I suggest we add a migration review step to the PR template." Teams that act on retrospective action items improve sprint over sprint. Teams that skip retros or treat them as complaint sessions stagnate. As a junior, your perspective is valuable — you notice inefficiencies that senior engineers have learned to tolerate.

**Kanban as an alternative.** Some teams use Kanban instead of Scrum. Kanban visualizes work on a board with columns like "To Do," "In Progress," "Review," and "Done." There are no fixed sprints — work flows continuously. Team members pull new work when they have capacity rather than being assigned tasks. Kanban works well for teams with unpredictable or maintenance-heavy workloads. As a junior, you will likely encounter both Scrum and Kanban. The principles are the same: break work into small pieces, limit work in progress, and focus on finishing before starting new work. The main difference is the timebox — Scrum imposes a rhythm through sprints, while Kanban relies on flow and work-in-progress limits to prevent bottlenecks.

### Building a Portfolio

Your portfolio is the single most important asset in your job search. A resume lists what you claim you can do. A portfolio proves it.

**Why a portfolio matters more than a resume.** Employers receive hundreds of resumes for every junior position. Most look the same: a list of technologies, a degree (or no degree), maybe some internships. A portfolio makes you real. When an interviewer can open your project, read your code, and see a live demo, they do not need to wonder whether you actually know what you are doing. The evidence is right there. One well-executed project can make up for a lack of formal education or professional experience. It is the great equalizer for self-taught developers.

**What makes a good project.** Three criteria. First, it solves a real problem — even a small one. A to-do app is boring. A to-do app that syncs across devices, sends reminders, and supports collaboration is interesting. The problem does not need to be original. It needs to be executed well. Second, it shows technical range. Use a database, build an API, handle authentication, deploy to a real server. Demonstrate that you can work across the stack. Third, it is complete. A half-finished project with broken features and no README hurts more than having no project at all. It signals that you do not finish what you start. A project that has a working demo, clean code, and documentation tells an employer: "This person ships."

**Project ideas.** If you are struggling for ideas, start with something you would actually use. A personal budgeting tool that categorizes expenses from a CSV bank export. A bookmark manager with tags, search, and browser extension. A habit tracker that sends weekly progress emails. A dashboard that shows your GitHub activity, pull request stats, and contribution streaks. A tool that scrapes a website you visit frequently and presents the data differently — movie ratings from a streaming service visualized by genre and year. The best projects are the ones you are motivated to maintain because you use them yourself. Another option is contributing to an existing open source project — more on that in the next section.

**How to document projects.** Every project needs a README. A good README includes: what the project does, how to run it locally, what technologies it uses, a link to a live demo, and a brief architecture overview. Include screenshots or a short video. If the project has an interesting technical challenge, explain how you solved it. This gives interviewers something to ask about. A well-documented project shows communication skills, which are just as important as coding skills.

**Anatomy of an effective README.** Open with a one-line description of what the project does. Follow with a screenshot or demo GIF — visual proof that it works. Then a table of contents for longer READMEs. The "Features" section lists what the project can do as bullet points. The "Tech Stack" section lists languages, frameworks, databases, and hosting with links. The "Setup" section gives step-by-step instructions to run the project locally — clone, install, configure environment variables, run migrations, start the server. The "Architecture" section provides a high-level overview with a diagram if possible. Finally, a "Roadmap" or "Future Improvements" section shows you think ahead. A README like this tells the reader that you care about the experience of other developers, which is exactly the attitude employers want.

**Code quality in portfolio projects.** Your portfolio code should reflect professional standards. Use consistent formatting — run a linter and formatter like Prettier or Black. Organize your code into clear modules or components. Remove dead code, commented-out blocks, and debugging console.log statements. Write meaningful commit messages. A reviewer who looks at your code should see evidence of discipline and craft. Sloppy code in a portfolio project sends the message that you will write sloppy code on the job.

**Writing good commit messages.** The commit history of a project tells a story. Use conventional commit format: `type(scope): description`. For example, `feat(auth): add password reset flow` or `fix(api): handle null response from payment gateway`. The message should complete the sentence "This commit will..." — so "add password reset flow" makes sense, while "fixes" does not. Keep the subject line under 72 characters. Add a blank line and a body for complex changes that explains the why, not the what. A reader should be able to understand the evolution of the codebase by scanning the commit log. This discipline signals that you think about the engineers who will maintain your code after you.

**Deployment and maintenance.** A project that exists only on your laptop might as well not exist. Deploy your projects to a public hosting platform so anyone can use them. Free tiers from Render, Railway, Vercel, Netlify, or Fly.io are sufficient for portfolio projects. Set up a simple CI pipeline — GitHub Actions is free for public repositories — that runs your tests on every push and deploys automatically when changes merge to main. A project that is deployed, tested, and maintained over several weeks signals reliability and discipline. If you fix bugs and add features over time, the commit history tells that story. A single green CI badge in the README says more about your engineering discipline than a paragraph of text.

**Portfolio project walkthrough.** Imagine you build a URL shortening service. Here is how you make it portfolio-ready. The README explains the problem (long URLs are hard to share, short URLs track clicks). It links to a live demo on Render. The tech stack is listed: Python FastAPI backend, PostgreSQL database, React frontend, deployed via GitHub Actions. The README includes a screenshot of the dashboard showing click analytics. The features section lists core functionality: create short URLs with custom slugs, click tracking with referrer data, expiration dates for links, an API for programmatic creation. The code is organized into `api/`, `models/`, `tests/`, and `frontend/` directories. The test directory has unit tests for the URL encoding logic and integration tests for the API endpoints. There is a `CONTRIBUTING.md` that welcomes pull requests. The commit history shows an evolution from initial scaffold through feature additions to bug fixes. When an interviewer asks about this project, you can talk about how you chose the encoding scheme, how you handled race conditions on slug generation, how you set up the CI pipeline, and what you learned from deploying and maintaining it for three months.

**Beyond the README.** A great portfolio project has more than just a README. A CONTRIBUTING file shows you welcome collaboration. A LICENSE file (MIT is the standard choice for portfolio projects) shows you understand open source conventions. A CHANGELOG tracks versions and notable changes. Issue templates and pull request templates show you have thought about how others interact with your project. A project wiki or architecture documentation shows depth. None of these are required, but each adds a signal that you think and act like a professional engineer.

**Quality over quantity.** Two or three solid, complete, well-documented projects will open more doors than ten half-finished experiments. Employers would rather see one project where you handled authentication, wrote tests, and deployed to production than ten repositories with three commits each and no README. Be selective. Invest your time in making a few projects excellent. When in doubt, ask yourself: "Would I be proud to show this to an interviewer?" If the answer is not a clear yes, put in more work before adding it to your portfolio.

### Open Source Contribution

Open source is not just about charity. It is one of the best ways to gain real-world experience, build a reputation, and get hired.

**Why contribute.** Open source projects are real software used by real people. When you contribute to one, you experience the full professional workflow: reading someone else's code, following contribution guidelines, participating in code review, and dealing with maintainers who have their own opinions and constraints. This is as close to a real job as you can get without being hired. Your contributions are public and permanent. They serve as a portfolio that any employer can inspect. And they put you in a community where opportunities — jobs, mentorship, collaboration — naturally arise. Many engineers have gotten their first job because a maintainer they worked with referred them or because an employer saw their contributions and reached out.

**How to start.** Do not start by trying to rewrite the codebase. Find beginner-friendly issues. Many projects label issues as "good first issue," "beginner friendly," or "help wanted." Read the contributing guidelines carefully — every project has different conventions for branching, commit messages, and pull request formatting. Join the project's communication channel (Slack, Discord, mailing list) and introduce yourself. Say you are looking for a first issue and ask for guidance. Most communities are welcoming to newcomers who show genuine interest. A typical message: "Hi, I am a new developer looking to make my first open source contribution. I noticed the issue about updating the README examples. Could I take that on? I have read the contributing guide and will follow the PR template."

**Start small.** Your first contribution should be something manageable. A documentation fix — correcting a typo, clarifying an unclear paragraph. A bug fix with clear reproduction steps. A small test addition. These teach you the contribution process without overwhelming you with code complexity. As you gain confidence, move to larger tasks: implementing small features, refactoring code, improving test coverage. Each contribution builds your understanding of the codebase and your relationships with the maintainers.

**Navigating the contribution process step by step.** Here is what the end-to-end process looks like. Find an issue that interests you. Comment on it saying you would like to work on it — this prevents duplicate work. Fork the repository to your GitHub account. Clone your fork locally. Create a branch following the project's conventions (often `git checkout -b fix/issue-123`). Make your changes. Write or update tests. Commit using the project's commit message style. Push the branch to your fork. Open a pull request from your fork to the original repository. Fill out the PR template if one exists. In the PR description, reference the issue number with `Closes #123` so GitHub links them. Wait for review. Address feedback — make requested changes, push new commits, and comment on the review threads. Once approved, a maintainer merges your PR. Delete your branch. Congratulations — you are now an open source contributor.

**Handling PR feedback in open source.** Open source reviews can be more direct than corporate reviews. Maintainers are often volunteers with limited time. They may leave short, to-the-point comments. Do not take brevity as rudeness. Address every comment. If you disagree, explain why with evidence. If a reviewer suggests a different approach, try it before arguing against it — you might learn something. If a PR sits without review for weeks, it is okay to politely bump the thread: "Hi, just checking if there is anything else needed on this PR." Do not bump more than once every two weeks. Respect that maintainers have their own priorities.

**Choosing a project to contribute to.** Pick a project you actually use or find interesting. If you use a library every day and notice a rough edge, that is a natural entry point. Look for projects with active maintainers — check that pull requests are reviewed and merged within a reasonable time. Read the issue tracker to understand the project's current priorities. Avoid projects that have stale issues, unanswered pull requests, or maintainers who seem burned out. A good first open source experience comes from a healthy, welcoming community. If the first project you try is not responsive, move on to another one.

**Where to find beginner-friendly projects.** GitHub's "Explore" page has a section for "Good First Issues." The website `goodfirstissue.dev` aggregates beginner-friendly issues across many projects. The `firstcontributions` repository on GitHub is designed specifically for first-time contributors with a step-by-step walkthrough. The `up-for-grabs` website lists projects that welcome new contributors. The `awesome-for-beginners` GitHub repository curates projects with good first issues. Search on GitHub for `label:"good first issue"` combined with your language of interest, like `label:"good first issue" language:python`. Start with projects that have explicit codes of conduct — they tend to have healthier, more welcoming communities.

**Building a GitHub presence.** Your GitHub profile is often the first thing an employer looks at. Make it count. Use a clear profile photo and a bio that says who you are and what you do. Pin your best repositories. Keep your contribution graph active — even if it is your own projects, regular activity signals that you are actively engaged. Write clean commit messages. Keep your repositories organized. A well-maintained GitHub profile is a passive job advertisement that works for you 24 hours a day.

### Resume and Online Presence

Your resume and online profiles are how employers find you and decide whether to interview you. They need to tell a coherent story about who you are and what you can do.

**What employers look for in junior resumes.** They are not looking for ten years of experience — you do not have it. They are looking for signals of competence and potential. A well-written projects section with links to live demos. A clear list of technologies you know, with honest proficiency levels — do not list every language you have heard of. List what you can actually use to build something. Evidence of learning — self-study, courses, bootcamps, open source contributions. Any non-tech work experience that demonstrates transferable skills: communication, teamwork, problem-solving, reliability. Do not leave work history off your resume just because it is not tech-related. A former restaurant manager has led teams, handled pressure, and resolved conflicts — those are real skills. A former retail worker has dealt with customers, managed inventory, and worked under time pressure. Frame every experience as preparation for engineering work: "I debugged inventory discrepancies" sounds like debugging code. "I coordinated with five departments to launch a promotion" sounds like cross-team collaboration.

**Structuring your resume.** Put your projects section near the top, before work experience. Each project entry should include the name, a one-line description, the technologies used, and links to the repository and live demo. Follow with skills — grouped into categories like languages, frameworks, tools. Then work experience, even non-tech roles, with bullet points describing accomplishments rather than duties. Education, if any, goes at the bottom. Keep it to one page. Use a clean, simple format. No fancy graphics or columns — applicant tracking systems cannot parse them.

**Writing effective bullet points for work experience.** For each role, write 3-5 bullet points. Each bullet should follow the format: "Action verb + what you did + the result or impact." Compare "Responsible for managing inventory" (duty) with "Reduced inventory discrepancies by 30% by implementing a weekly audit process" (accomplishment). For non-tech roles, use analogies that frame the experience in engineering terms. "Resolved customer complaints escalated by the front-line team" becomes "Triaged and resolved escalated issues, reducing repeat complaints by 20%." "Trained new employees" becomes "Onboarded and mentored 5 new team members, reducing their ramp-up time by two weeks." Every role has accomplishments — you just need to surface them.

**Tailoring your resume for each application.** A generic resume sent to every job is less effective than one customized for the role. Read the job description carefully. Identify the key technologies and responsibilities mentioned. If the role emphasizes React and TypeScript and your project uses those, move that project to the top of your projects section. If the description mentions experience with SQL, make sure your SQL skills are visible. Small adjustments take ten minutes and can significantly improve your response rate. Employers can tell when a resume was clearly written for a different role. Keep a master resume with everything you have done, and create a tailored version for each application by selecting the most relevant entries.

**Cover letters — when and how.** Cover letters are optional at most tech companies, but they can help when your background is non-traditional. A good cover letter is three paragraphs. First: introduce yourself and state the role you are applying for. Second: explain why you are interested in this specific company — mention a product, blog post, or engineering practice they have that you admire. Third: briefly tell your story — what led you to software engineering and why you are ready for this role. Keep it under 300 words. Do not repeat your resume — add context that the resume cannot convey. If a cover letter is optional and you have nothing specific to say, skip it. A generic cover letter is worse than none.

**LinkedIn profile.** LinkedIn is the default professional network in tech. If you are not on LinkedIn, many recruiters will not find you at all. Your profile should mirror your resume. Add a professional photo (a clear headshot, not a selfie). Write a headline that says what you do: "Junior Software Engineer | Python, React, PostgreSQL." Publish your projects with descriptions and links. Connect with people in the industry — classmates, colleagues, people you meet at meetups. Engage with content by commenting thoughtfully on posts. Do not spam. Building a network takes time, but it pays off when you need referrals for job applications.

**Optimizing your LinkedIn for recruiters.** Use keywords in your headline and summary that match the roles you want. If you want a backend role, include "backend," "API," "database" in your headline. Fill out the "Featured" section with links to your best projects and blog posts. Enable the "Open to Work" setting (you can hide this from your current employer if you are employed). Set your location to cities where you want to work. Follow companies you are interested in — engage with their content so you appear in their ecosystem. Request recommendations from people you have worked with — classmates, project partners, open source maintainers. A single recommendation that speaks to your technical abilities is worth more than a dozen connections.

**Technical blog.** Writing clarifies your thinking. When you write about a concept you have learned, you discover gaps in your understanding. When you publish that writing, you demonstrate expertise and communication skills. Start a blog on a platform like Dev.to, Hashnode, or your own site. Write about things you have built or learned. A tutorial on how you deployed your first application. A deep dive into a bug you fixed. A comparison of two technologies you evaluated for a project. The quality of your writing matters more than the frequency. One excellent article per month is better than four shallow ones.

**What to write about when you are starting out.** You do not need an original insight. Write about your learning journey. "How I built my first full-stack application and what I learned about database design" is a perfectly good topic. Write about bugs you fixed — documenting the debugging process teaches others and reinforces your own learning. Write comparisons — "React vs Vue: what I learned building the same app in both frameworks." Write tutorials that fill gaps you noticed in existing documentation. Write about tools and workflows — "My VS Code setup for Python development." The best topics are things you recently learned — you still remember the confusion and the aha moments, which makes your writing more relatable. Each article is also a portfolio piece that demonstrates communication skills.

**GitHub profile.** Beyond pinned repositories, make sure your GitHub profile is complete. A README for your profile (a special repository named after your username) can include a brief introduction, your technologies, and links to your blog and LinkedIn. Ensure your contribution email is set correctly so your commits are attributed to you. Clean up old, incomplete projects. If a repository is abandoned, archive it. A curated GitHub profile signals intentionality.

### Interview Preparation

Software engineering interviews are a skill in themselves. They test different things than real work does, and you need to prepare specifically for them.

**Types of technical interviews.** The process usually follows a sequence. A phone screen — a 15-30 minute call with a recruiter or hiring manager to verify basic qualifications and interest. Be prepared to summarize your background in under two minutes. Have a quiet space and a stable internet connection. A coding challenge — either a take-home assignment or a live platform like HackerRank or CoderPad. Take-home assignments typically give you 24-72 hours to complete a small application or solve a set of problems. Treat them like professional work: write tests, document your decisions, and clean up your code. One or more onsite rounds (or virtual equivalent) that include coding problems, system design discussion, and behavioral questions. Each round evaluates different aspects of your ability.

**The phone screen — making a strong first impression.** The recruiter is checking three things: does your background match the role, can you communicate clearly, and are you genuinely interested. Prepare a 60-second introduction that covers who you are, what you have built, and what you are looking for. Example: "I am a self-taught developer who built a full-stack expense tracking application using React and Django. I am looking for a junior role where I can contribute to a product team and continue growing as an engineer." Have questions ready for the recruiter too: "What is the team structure?" "What technologies does the team use?" "What is the interview process after this call?" Send a brief thank-you email after the call. Recruiters remember candidates who are prepared, professional, and enthusiastic.

**Take-home assignments — treating them like professional work.** If you receive a take-home assignment, treat it as a paid project even though it is unpaid. Start by reading the requirements carefully and asking clarifying questions if anything is ambiguous. Plan your approach before writing code. Write tests first — this shows you follow TDD even in assessment contexts. Commit early and often to show your process. Document your decisions in the README: why you chose certain technologies, what trade-offs you made, what you would improve given more time. When you submit, include instructions for running the code. Do not submit late. If the recruiter gave you 72 hours, submit in 48 — it signals that you are responsive and efficient. Many candidates fail take-homes not because their code is wrong but because they submitted something that does not run or is poorly documented.

**Coding interview prep.** The standard approach is to practice on LeetCode or a similar platform. Start with Easy problems. Focus on understanding patterns, not memorizing solutions. Move to Medium problems once you can solve an Easy in under 20 minutes. Hard problems are rarely asked at the junior level, but reaching them builds depth. Common topics: arrays, strings, hash maps, trees, graphs, dynamic programming, recursion. Dedicate time each day — even 30 minutes of practice compounds significantly over a month. Study the common patterns: two pointers, sliding window, breadth-first search, depth-first search, binary search, dynamic programming with memoization. When you encounter a new problem, categorize it by pattern before writing code. Practice speaking your thought process out loud while you code — in live interviews, how you think matters as much as the final answer. Time yourself. Do not spend more than 45 minutes on a single problem during practice. If you are stuck, look at the solution, understand it, and implement it from memory the next day. Spaced repetition works — revisit problems you solved a week ago and see if you can still solve them.

**A structured approach to solving coding problems in an interview.** When the interviewer presents a problem, do not start coding immediately. Follow this process. First, clarify the problem: ask about edge cases, input size, expected output format. "What happens if the array is empty?" "Is the input guaranteed to be sorted?" Second, state the brute force approach out loud and analyze its time and space complexity. Third, propose an optimized solution. Explain why it is better. Fourth, write code — clean, readable, with meaningful variable names. Fifth, walk through your code with an example input. Trace the execution line by line. Sixth, discuss test cases: normal case, edge case, error case. This structured approach demonstrates that you are methodical, which is exactly what interviewers want to see.

**System design for juniors.** Juniors are not expected to design Twitter or YouTube from scratch. But you should be able to design a simple system: a URL shortener, a chat application, a blog platform. Focus on the fundamentals: how to estimate traffic and storage, how to choose a database, how to structure an API, how to handle basic scaling (caching, load balancers, read replicas). Draw diagrams. Explain trade-offs. If you can walk through a simple design clearly and acknowledge where it would break under load, you will do well at the junior level.

**A sample system design walkthrough: URL shortener.** The interviewer asks you to design a URL shortener like bit.ly. Start by clarifying requirements: how many URLs per day (estimate 10 million), how long should the short URL be (6-7 characters), do we need analytics (yes). Estimate storage: 10 million new URLs per day times 500 bytes each equals about 5 GB per day, or 1.8 TB per year. Design the API: `POST /shorten` with the original URL in the body returns a short code; `GET /{short_code}` redirects to the original URL. Choose a database: PostgreSQL for persistent storage, Redis for caching frequently accessed URLs. Design the encoding: take a unique ID, encode it in base62 (a-z, A-Z, 0-9) to get a short string. Discuss scaling: add read replicas for the database, use a CDN for cache hit redirects, shard by short code prefix. Walk through a request: user submits a URL, server generates an ID, encodes it, stores the mapping in Postgres and Redis, returns the short URL. When someone visits the short URL, the server checks Redis first, then Postgres, and issues a 301 redirect. This level of detail shows you understand trade-offs and can think through a real system.

**Behavioral questions.** These questions assess whether you are someone the team wants to work with. The STAR method gives you a structured way to answer. Situation: set the context. Task: describe what you needed to accomplish. Action: explain what you did specifically. Result: share the outcome, ideally with numbers. Prepare stories for common questions: a time you resolved a conflict, a time you made a mistake, a time you took initiative, a time you worked under pressure. Write them down. Practice delivering them concisely. Each story should take 60-90 seconds to tell.

**Building your STAR story library.** Create a document with 5-7 stories that cover different scenarios. A story about technical problem-solving: "Our application was crashing under load during peak hours. I profiled the code, discovered an N+1 query in the dashboard endpoint, implemented eager loading, and reduced response time by 80%." A story about conflict resolution: "Two teammates disagreed on the database schema for a new feature. I proposed a meeting where each person presented their design with trade-offs. We agreed on a hybrid approach that used the best aspects of both. The feature shipped on time." A story about a mistake: "I accidentally merged a change that broke the staging environment. I immediately reverted, fixed the issue, added tests to prevent recurrence, and sent a postmortem to the team explaining what happened and what I learned." A story about leadership: "I noticed our onboarding documentation was outdated. I volunteered to rewrite it, interviewed three new teammates about what confused them, and produced a guide that reduced ramp-up time by two weeks." A story about persistence: "I spent three days debugging a memory leak that no one else could reproduce. I used heap snapshots, bisected recent commits, and eventually found a circular reference in event listeners. I fixed it and added a test that catches similar patterns." Having these stories ready means you never draw a blank during behavioral questions.

**Common junior questions.** "Tell me about yourself" — give a 60-second summary of your background, why you moved into software, and what you are looking for. "Why software engineering?" — be genuine. A story about building something and feeling the joy of creation is more compelling than "because it pays well." "Describe a challenging project" — use STAR, focus on a specific technical challenge and how you solved it. "Where do you see yourself in five years?" — they want to know you have thought about your growth, not that you have a precise five-year plan. "Do you have any questions for us?" — always say yes. Ask about the team, the tech stack, the onboarding process, or what success looks like in the first six months.

**Questions to ask the interviewer.** The interview is a two-way conversation. You are also evaluating whether the company is right for you. Prepare thoughtful questions. "What does the onboarding process look like for a new junior engineer?" "How is code review handled on the team?" "What is the ratio of senior to junior engineers?" "What is the most challenging technical problem the team is working on right now?" "How do you measure success for someone in this role?" Avoid questions about salary, benefits, or vacation in the early rounds — those are for the offer stage. Asking good questions shows you are thoughtful, curious, and serious about the role.

**Practice mock interviews.** Reading about interview techniques is not enough. You need to practice under realistic conditions. Find a friend or use a service like Pramp or interviewing.io. Simulate the full experience: time pressure, someone watching you code, explaining your reasoning out loud. The first mock will feel awkward. The tenth will feel natural. By the time you have your first real interview, you want the format to be familiar enough that you can focus on the problem, not the process.

**Common Interview Questions and Starter Answers**

**Why do you want to work here?** Research the company before the interview. Know what they build, who their customers are, and what technologies they use. Connect your answer to something specific: "I have been using your API for my side project and admired the documentation quality. I want to contribute to a team that values developer experience." Generic answers like "I love the culture" or "I have heard great things" are weaker than specific, genuine reasons. Spend 30 minutes before the interview reading the company blog, engineering blog, and recent product updates. Note one or two things that genuinely interest you and reference them in your answer.

**What is your biggest weakness?** This question tests self-awareness and honesty. Do not give a fake weakness that is actually a strength ("I work too hard"). Pick a real weakness you have identified and explain what you are doing to improve. "I used to struggle with estimating how long tasks would take. I started breaking work into smaller chunks and tracking my actual time against estimates. I am better now, but I still occasionally underestimate tasks, and I am learning to add buffer time for unknowns." The key is to show that you recognize the weakness and are actively working on it. An answer that ends with "and I am still working on it" is better than one that claims the weakness is fully resolved — the first is honest, the second sounds like you are not self-aware.

**Tell me about a time you had to learn something quickly.** The interviewer wants to see your learning process. Pick a concrete example: a new technology you picked up for a project, a domain concept you had to understand to fix a bug, a tool you learned to unblock yourself. Describe how you approached the learning: documentation, tutorials, building a small prototype, asking for help. Emphasize that you delivered despite being new to the area. This question is especially important for juniors — it signals that you can ramp up fast, which is what every team needs.

**Describe a time you disagreed with a teammate.** This tests your conflict resolution skills. Use the STAR method. The situation: you and a teammate disagreed on the approach for a feature. The task: you needed to reach a decision and ship the feature on time. The action: you scheduled a meeting to discuss both approaches with data — you compared performance benchmarks, maintenance cost, and development time. You reached a compromise that combined elements of both approaches. The result: the feature shipped on time, and the working relationship was stronger because you resolved the disagreement respectfully. This story shows you can handle disagreement without making it personal.

### The Job Search

Finding a job is a numbers game combined with a relationship game. You need both volume and targeted effort.

**Where to find jobs.** LinkedIn is the largest source of tech job listings. Set up job alerts for junior and entry-level software engineer positions. Indeed aggregates listings from many sources. Company career pages are underused — if you know you want to work somewhere, check their careers page directly. Many companies post jobs only on their own site. Networking referrals are the most effective channel. An internal referral dramatically increases your chances of getting an interview. Attend meetups, conferences, and online community events. Talk to people. Ask about their work. Do not ask for a job directly — build relationships first.

**Building a target company list.** Do not apply to every job you see. Create a list of 20-30 companies you would actually want to work for. Research each one: what they build, their engineering culture, their tech stack, their size and stage (startup, mid-size, enterprise). Rank them by preference. For your top 5-10, invest extra effort: tailor your resume, write a specific cover letter, try to get a referral through your network. For the rest, apply with your standard resume. This approach ensures you do not miss opportunities at the companies you care about most while still maintaining volume. As you progress through interviews, you will have choices — and choices give you leverage.

**Entry-level vs. junior roles.** Some companies use "entry level" and "junior" interchangeably. Others distinguish: entry level is for new graduates or career changers with no professional experience; junior means 1-2 years of experience. Apply to both. Do not filter yourself out by reading the "requirements" section too literally. If a job asks for 3 years of experience and you have 1, apply anyway. The requirements list is often a wishlist, not a hard cutoff. Let the employer decide if you are qualified. Many junior job descriptions are written for the ideal candidate who does not exist — hiring managers routinely compromise on requirements when they find a candidate who shows potential and a willingness to learn.

**Internships.** Internships are not just for college students. Many companies offer internships to career changers and self-taught developers. An internship provides structured onboarding, mentorship, and a path to a full-time offer. The pay is lower than a full-time salary, but the learning velocity is higher. If you are struggling to land a full-time role, consider internships as a stepping stone. Some programs, like the Microsoft LEAP apprenticeship or Google's apprenticeship program, are specifically designed for career changers. An internship also gives you a chance to test a company before committing full-time. Three months is enough to know whether you would be happy working there.

**Networking.** Networking feels uncomfortable when you are starting out. You feel like you have nothing to offer. That is not true. You have curiosity, enthusiasm, and a willingness to learn. Experienced engineers enjoy talking to people who are genuinely interested in the craft. Go to meetups. Join Discord servers for technologies you use. Participate in discussions. Ask thoughtful questions. Help others when you can — even beginners can answer questions from even more junior people. The relationships you build will open doors that cold applications never will.

**Where to network.** Local meetup groups on Meetup.com focused on your tech stack or interests. Conferences — many offer discounted or free tickets for students, career changers, or underrepresented groups. Online communities like the r/cscareerquestions subreddit, Dev.to, Hacker News threads, and technology-specific Discord servers. Twitter (or X) — follow engineers you admire, engage with their content, share what you are learning. LinkedIn — connect with people after meeting them, engage with their posts, share your own content. Networking does not mean collecting business cards. It means building genuine relationships over time. Send a message after meeting someone: "Hey, I enjoyed our conversation about testing strategies at the meetup. I tried your suggestion about property-based testing and it caught a bug I would have missed. Thanks!"

**The informational interview.** This is one of the most effective networking tools. Reach out to an engineer at a company you are interested in and ask for 15-20 minutes of their time. Say: "I am exploring software engineering as a career and would love to hear about your experience at [company]. I have prepared a few specific questions." During the conversation, ask about their day-to-day work, what they wish they had known as a junior, what the engineering culture is like, and what the company looks for in new hires. Do not ask for a job. Do not ask for a referral unless they offer. At the end, thank them and ask if there is anyone else they recommend you talk to. Afterward, send a thank-you note. If you later apply to that company, you can mention your conversation in the application. Most engineers are happy to do informational interviews — they remember what it was like to be starting out.

**Tracking applications.** When you send out dozens of applications, you will lose track if you do not have a system. Use a spreadsheet to track each application: company name, role, date applied, where you found the listing, whether you have a referral, and the current status (applied, screening, interview scheduled, rejected, offer). Note any follow-up actions — "send a thank-you email after the interview," "follow up in one week if no response." A tracking system turns a chaotic job search into a manageable pipeline. It also helps you identify which channels are working. If you are getting no responses from cold applications on LinkedIn but every referral leads to an interview, you know where to focus your energy.

**Setting weekly goals for the job search.** A job search without targets drifts. Set concrete weekly goals: apply to 10 jobs, send 5 networking messages, complete 2 LeetCode problems, read 1 engineering blog post from a target company. At the end of each week, review your progress and adjust. If you are getting interviews but not offers, shift more time to interview prep. If you are not getting interviews, shift more time to portfolio improvement and networking. Treating the job search like a project with weekly sprints keeps you moving forward even when results are slow.

**Managing your pipeline strategically.** When you get your first interview invitation, do not stop applying. Continue sending applications until you have a signed offer. An interview that goes well does not guarantee an offer — candidates get rejected at the final stage every day. Maintaining a full pipeline protects you from the emotional crash of a near-miss. It also gives you leverage: when you have multiple interviews scheduled at the same time, you can accelerate timelines and compare offers. Be transparent with recruiters: "I am interviewing with other companies and would love to make a decision by [date]. Is it possible to expedite your process?" Most recruiters will work with you.

**Follow-ups and thank-you notes.** After an interview, send a brief thank-you email within 24 hours. Mention something specific from the conversation to show you were listening. If you discussed a technical topic, you might add a short follow-up on that discussion. If you have not heard back after a week following an interview, send a polite check-in. Keep it professional and concise. Do not pester. One follow-up per week is the maximum before you risk annoying the recruiter.

**Rejection is normal.** You will apply to dozens of jobs. You will hear nothing from most. You will get a few interviews. Some will go well and still result in rejection. This is not a reflection of your worth or your potential. The hiring process is noisy. Luck plays a role. The candidate who gets the offer is often not the most qualified — they are the one whose resume landed at the right time, with the right recruiter, on a good day. Your job is to increase your surface area for luck by applying consistently, improving your materials based on feedback, and not stopping. Every rejection is a data point. If you are consistently getting rejected after interviews, identify the pattern and work on it. If you are not getting interviews at all, improve your resume and portfolio.

**Learning from rejection.** When you receive a rejection, you can politely ask for feedback. "Thank you for letting me know. Would you be willing to share any feedback that might help me in future interviews?" Some recruiters will respond with useful information: "Your technical skills were strong, but we were looking for someone with more experience in distributed systems." Others will give generic responses or none at all. Take whatever feedback you get seriously. If two different interviewers mention that your communication during coding problems was unclear, that is a pattern to address. If no one gives feedback, record your own assessment of what went wrong while it is still fresh. Over time, you will get better at self-evaluation.

**Negotiating your first offer.** Many juniors accept the first offer without negotiation. Do not leave money on the table. When you receive an offer, thank the recruiter and ask for 24-48 hours to review it. Research market rates for junior engineers in your area using sites like Levels.fyi or Glassdoor. Consider the full compensation package: base salary, equity, signing bonus, annual bonus, benefits (health insurance, 401k matching, education stipends, remote work flexibility). Sometimes a lower base salary is offset by strong equity or benefits.

When negotiating, be specific. "I am very excited about this opportunity, and based on my research, the market rate for this role in this area is around X. Is there flexibility to move the offer closer to that number?" If they cannot move on salary, ask about alternative forms of compensation: a signing bonus, a guaranteed performance review in six months instead of twelve, additional equity, or a budget for conferences and courses. The worst they can say is no. Most recruiters expect some level of negotiation and will not rescind the offer for asking. But do not be aggressive or entitled. Be professional and appreciative. Remember that the relationship continues after negotiation — you will work with these people.

**Deciding between offers.** If you are fortunate enough to have multiple offers, do not just compare salary. Consider the growth opportunity: will you learn more at a smaller company where you wear many hats, or at a larger company with structured mentorship? Consider the team: did you enjoy talking to your potential manager and teammates? Consider the product: do you find the problem domain interesting? Consider the commute or remote policy. Consider job security: is the company well-funded or profitable? A decision matrix — listing each factor with a weight and scoring each offer — can help you make an objective comparison. The best offer is not always the one with the highest salary. It is the one that sets you up for the most growth in your first year.

### First 90 Days on the Job

You got the offer. You accepted. Now the real learning begins. The first 90 days set the tone for your entire tenure at a company.

**What to expect as a new junior engineer.** You will feel overwhelmed. There will be a codebase with thousands of files, a deployment process you do not understand, domain concepts you have never heard of, and a team that seems to move at lightning speed. This is normal. Every junior engineer goes through it. The goal of the first 90 days is not to be productive at the same level as your teammates. It is to become productive enough to take on small tasks independently. Think of it as a probationary period where the team is investing in you. They expect you to be slow at first. They expect you to have questions. They do not expect you to ship features on day one.

**Day one — environment setup.** Your first day will involve setting up your development environment. The team should have documentation for this. Follow it exactly. If the docs say install Node 18, install Node 18 — not 20. If something does not work, document the issue and the fix so you can contribute back to the setup guide. Once the environment is working, run the test suite and make sure it passes. Then deploy the application locally. Make a tiny change — change a button color or add a log statement — and verify you can see the effect. This validates that the entire development loop works: edit, build, test, run. By the end of day one, you should be able to make a change and see it in your local environment.

**First week — orientation and small wins.** Your first week is about orientation. Meet your teammates. Get access to all the systems: code repository, project management, CI/CD, monitoring, documentation. Read any onboarding materials. Set up your development environment if you did not complete it on day one. Find the simplest open ticket — a typo fix, a documentation update, a tiny bug — and fix it. Merge your first pull request. That first merge, even if it is a one-line fix, is psychologically important. It proves you can go through the entire workflow: branch, change, PR, review, merge. After that first merge, the process is familiar. Every subsequent contribution is just a variation on the same workflow.

**Learning the codebase, tools, and team practices.** Start by understanding the big picture. Ask for an architecture diagram. Read through the README and any onboarding documentation. Set up the development environment on your machine. Run the tests. Break something on purpose and fix it. Pick one small area of the codebase and trace through a request from the user interface to the database and back. Read the commit history to understand how the codebase evolved — messages like "add payment retry logic" tell you more about design decisions than the code alone. Learn the team's tools: the project management system (Jira, Linear, Trello), the communication channels (Slack, Discord), the CI/CD pipeline. Each day, you should understand a little more than you did the day before. Keep a running document of everything you learn — architecture notes, setup steps, helpful commands, common gotchas. That document will be invaluable not just for you but for the next junior engineer who joins after you.

**The learning log technique.** Create a document called "Learning Log" or "Onboarding Notes." Every day, write down three things: what you learned, what confused you, and what you accomplished. At the end of each week, review the log and note how much more you understand than you did on day one. This practice serves multiple purposes. It reinforces your learning through writing. It gives you material to discuss in one-on-ones with your manager. It becomes a resource you can reference when you encounter the same problem again. And when you eventually onboard a new junior engineer, you can hand them your learning log as a head start. A typical entry: "Day 5 — Learned how our deployment pipeline works: git push triggers GitHub Actions, which runs tests, builds a Docker image, pushes to ECR, and deploys to ECS. Confused about how environment-specific configuration is handled — need to ask about .env files vs. AWS Secrets Manager. Accomplished: fixed a broken link in the footer and deployed my first PR."

**Asking good questions.** You will have many questions. The skill is knowing when to ask and when to figure it out yourself. Before asking, try to find the answer on your own. Search the codebase. Check the documentation. Look at similar implementations. If you cannot find the answer in 15-30 minutes, ask. When you ask, provide context: what you are trying to do, what you have tried, what you expected to happen, and what actually happened. This makes it easy for your teammate to help you quickly. Keep a document of questions you have asked and their answers. You will likely ask the same thing twice, and the document prevents that.

**The art of the debugging question.** When you are stuck on a bug and need help, write out the full context before you ask. What is the expected behavior? What is the actual behavior? What are the steps to reproduce? What have you already checked? What do the logs say? Does the bug happen locally, in staging, or in production? Does it affect all users or just some? When did it start happening? A well-structured debugging question often gets answered within minutes because the person helping you can jump straight to the likely cause. A vague question like "the login is broken" requires a back-and-forth that wastes everyone's time. Write your debugging question as if you are filing a bug report — because you are.

**Taking ownership of your onboarding.** Do not wait for your manager or teammates to feed you tasks. Be proactive. Read the backlog of tickets and find something small you can start on. Shadow a teammate during their work. Ask if you can pair on a task to learn the workflow. Volunteer to write documentation for an area that has none — it forces you to understand it deeply, and the team will appreciate it. The engineers who succeed in their first 90 days are not the ones who know the most on day one. They are the ones who take initiative, ask for help when stuck, and make their teammates' lives easier.

**Pair programming as a learning tool.** Pair programming is one of the fastest ways to learn a codebase. In a pairing session, one person writes code (the driver) while the other reviews each line in real time (the navigator). When you are the navigator, you learn by watching an experienced engineer solve problems. When you are the driver, you get immediate feedback on your approach. Ask your teammates if you can pair for an hour on a task they are working on. Most will say yes. During the session, ask questions: "Why did you structure the function this way?" "What does this error mean?" "How did you know to check that log?" Do not try to pair all day — it is mentally exhausting. An hour or two is enough. After the session, write down what you learned and review it the next day.

**Building relationships with teammates.** Software engineering is a team sport. Your technical skills matter, but your ability to work with others matters more. Learn your teammates' names and what they work on. Schedule one-on-one meetings with each person in your first few weeks. Ask them about their background, what they are working on, and what they wish they had known when they started. Offer to help with small tasks. Be reliable — when you say you will do something, do it. Be pleasant to work with. Teams will tolerate a mediocre coder who is a great teammate far longer than a brilliant coder who is difficult to work with.

**The one-on-one meeting with your manager.** This is your most important recurring meeting. Come prepared. Write an agenda with these sections: wins (what you accomplished since the last meeting), challenges (what you are stuck on), learning (what you are learning and what you want to learn next), and feedback (any concerns or suggestions). Share the agenda with your manager before the meeting so they can prepare too. During the meeting, ask for specific feedback: "Is there anything I should start doing, stop doing, or do differently?" Ask about growth: "What skills should I focus on developing this quarter?" Your manager wants you to succeed — a well-run one-on-one makes it easy for them to help you.

**Setting expectations with your manager.** In your first week, ask your manager: "What does success look like in the first 30, 60, and 90 days?" Get specific answers. "Be able to fix bugs independently by day 60." "Ship one feature by the end of the quarter." "Write tests for the module you are assigned to." Write these down and review them regularly. Schedule regular one-on-one meetings with your manager — weekly is typical. Come prepared with an agenda: what you accomplished, what you are stuck on, what you plan to work on next. Ask for feedback explicitly. Do not wait for your manager to offer it.

**The imposter syndrome is real.** You will look at your teammates and feel like they are way ahead of you. They know things you do not know. They type faster, debug faster, and seem to understand everything. This is not because they are fundamentally different from you. It is because they have been doing this longer. Every single engineer you admire has felt exactly what you are feeling. The secret is that the feeling never fully goes away — it just shifts to new topics as you grow. You will always be a beginner at something. That is not a weakness. That is the nature of a field that changes constantly. Embrace it.

**Strategies for coping with imposter syndrome.** When the feeling hits, go back to facts. Look at your learning log and see how much you have learned in the past week or month. Remind yourself that your team hired you because they believe you can do the job — they have more information about what the job requires than you do. Talk to a trusted teammate about how you feel — you will likely discover they felt the same way when they started. Focus on progress, not perfection. You do not need to know everything. You need to know enough to take the next step. When you feel like a fraud, ask yourself: "Did I add value today?" If the answer is yes — even a small yes — you are doing your job.

**Time management and focus.** Software engineering requires long stretches of focused concentration. Context switching — jumping between chat, email, code, and meetings — is the enemy of deep work. Protect your focus time. Block out periods on your calendar for coding without interruptions. Batch meetings together if possible. Use techniques like the Pomodoro method (25 minutes of focused work followed by a 5-minute break) to maintain momentum. Learn to say no to distractions politely. When you are in flow, close Slack and email notifications. Your most valuable output as an engineer comes from uninterrupted thinking time, not from being available every minute. Guard that time carefully.

**The first bug fix — a walkthrough.** Your first real task might be fixing a bug. Here is how to approach it. Reproduce the bug locally first — if you cannot reproduce it, you cannot fix it. Read the error message carefully. Error messages tell you what went wrong and often where. Search the codebase for the relevant code. Add logging or use a debugger to trace the execution. Identify the root cause — not just the symptom. A button that does nothing might be broken because the event handler references a nonexistent function, but the root cause might be a refactoring that renamed the function without updating all callers. Fix the root cause, not the symptom. Write a test that would have caught the bug. Run the existing tests to make sure your fix does not break anything. Open a PR with a clear description of the bug, the root cause, and the fix. This process — reproduce, diagnose, fix, test, submit — is the fundamental unit of engineering work. Master it, and you can handle anything.

**Writing your first postmortem.** When something goes wrong in production — a server goes down, a bug affects users, a deployment breaks — your team will write a postmortem. A postmortem is a document that describes what happened, why it happened, what was done to fix it, and what will be done to prevent it from happening again. The tone is blameless. The goal is not to find who caused the incident but to understand the systemic weaknesses that allowed it. When you contribute to a postmortem, focus on facts: timelines, actions taken, observations. Suggest preventive measures: a missing test, a monitoring gap, a deployment process improvement. Writing good postmortems is a skill that distinguishes professional engineers. It shows that you think about reliability, not just features.

**The power of saying "I do not know."** One of the hardest things for new engineers is admitting they do not know something. They worry it makes them look incompetent. The opposite is true. Saying "I do not know, but I will find out" is a sign of confidence and intellectual honesty. It is far better than pretending to understand and making a mistake later. When a senior engineer explains something and you do not fully grasp it, say so. Ask them to rephrase or go slower. The best engineers ask the most clarifying questions, not the fewest. Every time you say "I do not know" and then learn the answer, you become a little bit better than you were before. Cultivate the habit early — it will serve you throughout your career.

### Beyond the First Job

Your first job is the beginning, not the destination. The career ahead of you spans decades. Think about how you want to grow.

**Continuing to learn.** The technologies you use in your first job will likely not be the ones you use in five years. Languages, frameworks, and tools change. The fundamentals — how to think, how to debug, how to design systems, how to work with people — do not change. Invest in the fundamentals. Learn new technologies, but do not chase every trend. Pick one or two areas to go deep on: a language, a domain (backend, frontend, data, infrastructure), a type of system. Depth makes you valuable. Breadth comes naturally over time as you encounter different problems.

**Learning in public.** Sharing your learning journey publicly compounds your growth. When you write about a topic, you learn it more deeply. When others read what you write, they give you feedback and corrections. When you build a reputation as someone who learns and shares, opportunities come to you. "Learning in public" does not mean you need to be an expert. Write about what you are currently learning, not what you already mastered. A series of posts titled "Learning about message queues by building one" is more valuable to readers and to your growth than a polished post that summarizes everything you know. The practice of explaining concepts to others forces you to fill gaps in your own understanding. Start small — a single blog post, a Twitter thread, a talk at a local meetup. The habit matters more than the audience size.

**Finding mentors.** A mentor accelerates your growth by years. A good mentor does not give you answers — they ask questions that help you think more clearly. They share their experience without expecting you to repeat their path. They tell you the truths that are uncomfortable to hear. Find mentors at work: senior engineers, your manager, experienced teammates. Find mentors outside work: people you meet at conferences, in online communities, through open source. Ask for specific advice, not general mentorship. "Can you review my design for this feature?" is a better ask than "will you be my mentor?" Let mentoring relationships develop organically. Be a good mentee: show up prepared, act on the advice you receive, and express gratitude. Mentors are more likely to invest time in someone who demonstrates growth from their guidance.

**Structured learning paths after your first year.** After you have settled into your first job, consider pursuing more structured learning. A deep-dive course on a topic relevant to your work, like distributed systems or database internals. A certification if your industry requires it (cloud certifications for infrastructure roles, for example). A side project that teaches you a new paradigm — building something in a different language or framework than you use at work. Reading a foundational book in computer science like "Designing Data-Intensive Applications" or "The Pragmatic Programmer." Set aside a few hours each week for deliberate learning. Without structure, it is easy to let work consume all your mental energy and stop growing. Your company may have a learning budget — use it for books, courses, or conference tickets.

**Giving back.** As you grow, you will become the person that newcomers look up to. Remember what it felt like to be a beginner. Pay it forward. Answer questions from juniors. Review code from interns. Write documentation that you wish had existed when you started. Speak at a meetup about your journey. Teaching is the best way to solidify your own understanding, and the community grows stronger when experienced engineers support newcomers. You do not need to be a senior engineer to mentor someone. If you are one year ahead of someone else, you can help them navigate the year you just completed.

**Starting to mentor.** When you are six months into your first job, you already know things that someone just starting does not. Offer to pair with an intern on their first PR. Answer questions in the team's onboarding channel. Write down the tips and tricks you have learned about the codebase and share them. The act of articulating what you know forces you to organize your thoughts and identify gaps in your own understanding. Mentoring also builds your reputation as someone who is helpful and invested in the team's success. When promotion conversations happen, your manager will remember that you contributed to the growth of others.

**Knowing when to stay and when to leave.** Your first job may be amazing or it may be mediocre. Either way, you should plan to stay at least one year. Anything less raises questions in future interviews. But do not stay past the point where you have stopped growing. Signs that it is time to leave: you are no longer learning new things, you dread going to work, your manager does not support your growth, the company is in decline, you have been passed over for promotion without clear reasons. When you leave, do so professionally. Give notice. Finish your tasks. Write documentation. Your reputation follows you.

**Leaving well.** When you decide to move on, give at least two weeks notice, and more if you can. Offer to help with the transition: write documentation, train the person who will take over your work, or document open tasks and their status. Do not burn bridges — even if you are leaving because the job was bad. The software industry is small. People change companies. Your former manager might end up at a company you want to join later. Your former teammates will be asked about you in reference checks. Leave on good terms, and your network will remain strong. Write a thoughtful farewell message in the team channel. Say thank you to the people who helped you grow. The way you leave is remembered as much as the way you worked.

**Building your professional network.** Your network is one of your most valuable career assets. Stay in touch with former teammates and managers — send a message every few months to check in. Attend meetups and conferences regularly, even after you have a job. Join online communities in your area of interest. Be helpful: answer questions, share resources, introduce people who might benefit from knowing each other. When you need advice, a referral, or a recommendation, your network is there. But do not treat it transactionally. Build genuine relationships by being curious about others and generous with your knowledge. The most powerful networks are built over years, not weeks. Start now.

**Building your personal brand.** As you gain experience, your reputation becomes a career asset. Write about what you learn. Speak at meetups or conferences. Contribute to open source. Answer questions on Stack Overflow or in community forums. A visible engineer gets opportunities — job offers, speaking invitations, consulting requests — without applying for them. You do not need to be a Twitter influencer or a conference keynote speaker. A consistent blog with thoughtful content and a GitHub profile with meaningful contributions is enough to establish you as someone who knows their craft.

**Career paths and specializations.** After your first year or two, you will start to discover what you enjoy most. Some engineers love the frontend — crafting interfaces, optimizing animations, designing component systems. Others prefer the backend — building APIs, designing databases, handling distributed systems. Some go into data engineering, infrastructure and DevOps, mobile development, or machine learning. Each path requires different skills and offers different rewards. You do not need to decide now. Try different areas. Your early career is for exploration. The [Types & Specializations](types-and-specializations.md) document covers the major paths in detail.

**Individual contributor vs. management track.** As you gain experience, you will face a choice: stay on the individual contributor (IC) track and keep coding, or move into engineering management and lead teams. Neither is better. The IC track lets you stay deep in technology, architect systems, and mentor other engineers. The management track lets you shape team culture, grow people, and influence the direction of products. Some engineers switch between tracks over the course of their career. Your first year or two is too early to decide — focus on becoming a strong IC first. But be aware that the choice exists, and it is not permanent. You can be a manager for a few years and return to being an IC, or stay on the IC track for your entire career.

**Financial and career planning.** As your income grows, your financial decisions become more important. Live below your means. Build an emergency fund of 3-6 months of expenses. Invest early — compound interest is the most powerful force in personal finance. Max out any retirement matching your employer offers. Consider the total comp of your next move, not just the salary — equity and bonuses can be significant at many companies. Keep learning about personal finance. Many engineers are well-compensated but financially unprepared because they never learned the basics. A few hours of research early in your career can save you decades of financial stress.

**Navigating performance reviews.** Most companies have quarterly or annual review cycles. Your performance review determines your raise, promotion, and sometimes your continued employment. Do not wait for the review to prepare. Keep a "brag document" — a running list of your accomplishments, positive feedback from teammates, and metrics that show your impact. Update it weekly. When review time comes, you will have a comprehensive record instead of trying to remember everything you did in the past six months. During the review, focus on outcomes, not activities. "Shipped the notification system that reduced user churn by 15%" is stronger than "worked on the notification system." Ask for specific, actionable feedback: "What should I do differently to reach the next level?" If you disagree with a rating, present evidence calmly and professionally. Most managers will work with you if you make a data-backed case.

**The first promotion.** Your first promotion — from junior to mid-level — is a major milestone. It typically comes after 1-2 years. To earn it, you need to demonstrate that you can take a task from specification to deployment with minimal supervision. You write tests without being reminded. You participate actively in code review. You start to help other juniors. You understand the domain well enough to anticipate problems. Ask your manager what the criteria are for promotion at your company. Different companies use different frameworks. Some require you to have led a project. Others look for breadth across the codebase. Others value mentorship. Know the target, and work toward it deliberately. If you have been in role for two years without a promotion conversation, it is time to ask for one. Your manager may not proactively initiate it — especially at fast-moving companies where priorities shift constantly.

**The career is a marathon, not a sprint.** You will have periods of rapid growth and periods where you feel stuck. You will have jobs you love and jobs you tolerate. You might leave the industry and come back. You might become a manager, a principal engineer, a founder, or something else entirely. None of these paths is better than the others. The only thing that matters is that you keep learning, keep building, and keep finding joy in the work. The fact that you have made it this far — from the first line of code to reading about professional development — means you already have the most important quality: the willingness to grow. Trust that.

**Staying healthy in a demanding career.** Software engineering can be physically and mentally demanding. Protect your health as you would protect your career. Set boundaries on working hours — occasional crunch is normal, but consistent overtime is a management failure, not a badge of honor. Take care of your body: sit-stand desk, regular exercise, eye breaks every 20 minutes. Protect your sleep — sleep deprivation destroys cognitive performance, which is your primary tool as an engineer. Take all your vacation days. Disconnect fully during time off. A burned-out engineer is not productive and not happy. The best engineers are the ones who sustain their performance over decades, not those who burn out in two years. Learn to recognize the early signs of burnout: persistent fatigue, cynicism about work, declining performance. If you notice these, talk to your manager, adjust your workload, and prioritize recovery. Your career is long. Treat your body and mind like they need to last.

## Glossary

| Term | Definition |
|------|------------|
| Unit Test | A test that verifies a single function, method, or component in isolation |
| Integration Test | A test that verifies that multiple components or systems work together correctly |
| End-to-End Test | A test that simulates a real user interacting with the full application |
| TDD | Test-Driven Development — writing a failing test before writing the production code, then making it pass |
| CI/CD | Continuous Integration and Continuous Deployment — automated testing and release pipelines |
| Code Review | A process where developers review each other's code changes before merging |
| Pull Request | A proposed code change submitted for review and discussion before merging |
| Agile | A software development methodology focused on iterative delivery, collaboration, and responding to change |
| Scrum | A specific Agile framework using fixed-length sprints, standups, and retrospectives |
| Sprint | A fixed time period (usually 1-2 weeks) during which a team commits to completing a set of work |
| Standup | A daily team meeting where members share progress, plans, and blockers |
| Retrospective | A meeting at the end of a sprint to discuss what went well, what could improve, and what to change |
| STAR Method | Situation, Task, Action, Result — a structured format for answering behavioral interview questions |
| Mock Interview | A practice interview simulation designed to prepare for real interviews |
| Referral | A recommendation from an employee that can significantly increase the chance of getting an interview |
| Portfolio | A curated collection of projects demonstrating a developer's skills and capabilities |
| Imposter Syndrome | The persistent feeling of being unqualified despite evidence of competence |
| Onboarding | The process of integrating a new employee into a team and teaching them the tools and practices |
| One-on-One | A regular meeting between a manager and a direct report to discuss work, growth, and feedback |
| Pair Programming | Two developers working together at one workstation — one writing code, the other reviewing in real time |
| Dependency Injection | A design pattern where dependencies are passed into a function or object rather than created internally |
| Mock | A fake object that simulates the behavior of a real dependency for testing purposes |
| Stub | A minimal implementation of a dependency that returns fixed data for testing |
| Code Coverage | A metric measuring what percentage of code is executed during testing |
| Kanban | A workflow management method that visualizes work on a board with continuous flow rather than fixed sprints |
| Trunk-Based Development | A branching strategy where developers merge small changes to the main branch frequently |
| Phone Screen | An initial brief interview (usually by phone or video) to verify basic qualifications |
| Behavioral Question | An interview question that asks about past behavior to predict future performance |
| Take-Home Assignment | A coding task given to candidates to complete on their own time as part of the interview process |
| Applicant Tracking System | Software that parses and filters resumes automatically before a human sees them |
