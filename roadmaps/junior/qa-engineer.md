# Junior QA Engineer

## Description

What a junior QA engineer should know — manual testing fundamentals, bug reporting, test case design, and basic automation concepts.

## Prerequisites

- Basic computer literacy — files, folders, browser developer tools

## Learning Path

### Testing Fundamentals

- `[must-know]` What is software testing — purpose, goals, quality attributes
- `[must-know]` Manual testing — exploratory testing, ad-hoc testing
- `[must-know]` Test case design — happy path, edge cases, negative scenarios
- `[must-know]` Bug reporting — clear reproduction steps, expected vs actual, environment info
- `[good-to-know]` Smoke testing vs sanity testing vs regression testing
- `[good-to-know]` Test documentation — test plans, test suites, checklists

### Types of Testing

- `[must-know]` Functional testing — verifying features work as specified
- `[must-know]` UI/UX testing — layout, responsiveness, usability
- `[must-know]` Cross-browser testing — Chrome, Firefox, Safari, Edge
- `[must-know]` Mobile testing basics — screen sizes, touch interactions
- `[good-to-know]` API testing — sending requests, verifying responses
- `[good-to-know]` Performance testing basics — load time, responsiveness
- `[nice-to-have]` Security testing basics — input validation, auth checks

### Tools

- `[must-know]` Bug tracking — Jira, Linear, GitHub Issues (creating, triaging, updating)
- `[must-know]` Browser DevTools — Console, Network, Elements tabs
- `[good-to-know]` Test management — TestRail, Zephyr, or spreadsheets
- `[good-to-know]` API clients — Postman, Insomnia for manual API testing
- `[nice-to-have]` Screen recording and screenshot tools — Loom, Snagit

### Automation Awareness

- `[good-to-know]` What test automation is and when it's worth it
- `[nice-to-have]` Writing basic assertions — comparing expected vs actual values
- `[nice-to-have]` Simple UI automation scripts — Playwright Codegen, Selenium IDE

### Communication

- `[must-know]` Writing clear, concise bug reports
- `[must-know]` Communicating severity vs priority
- `[good-to-know]` Working with developers — reproducible steps, providing logs
- `[good-to-know]` Participating in sprint ceremonies — planning, review, retro

### Domain Knowledge

- `[good-to-know]` Understanding the product domain — user workflows, business rules
- `[good-to-know]` Reading technical specifications and user stories
