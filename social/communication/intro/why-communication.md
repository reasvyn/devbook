# Why Communication?

## Description

Communication is the exchange of information between people. For developers, it determines whether code reviews are productive, whether docs are useful, whether requirements are clear, and whether teams trust each other.

## Content / Material

### What Is Communication?

Communication is the transmission of information from sender to receiver. It seems simple, but every transmission is filtered through context, knowledge, biases, and expectations.

**The conveyance problem:** What you say is not what I hear. Every message passes through two filters — what you meant to say and what I understood. The gap between these is where most communication failures live.

### Why Developers Should Care

**Software development is a communication problem disguised as a technical problem.** Most project failures trace to communication breakdowns, not technical shortcomings.

- **Requirements** — unclear communication produces the wrong product
- **Code reviews** — poor feedback creates resentment, not improvement
- **Documentation** — the best API is useless if nobody can understand how to use it
- **Remote work** — async communication is the backbone of distributed teams
- **Career growth** — the most technically skilled developer who can't communicate will be passed over for senior roles

**The career multiplier.** Technical skill determines your floor. Communication skill determines your ceiling.

### Communication in Practice

**Written communication dominates software development:**

- RFCs and design documents
- Pull request descriptions and comments
- Slack messages and emails
- Documentation and READMEs
- Commit messages
- Incident reports and post-mortems

**Principles of effective developer communication:**

| Principle | Practice |
|-----------|----------|
| **Know your audience** | Write for the reader's knowledge level, not yours |
| **State the context first** | Start with the why, then the what, then the how |
| **Be concise** | Remove every word that doesn't add information |
| **Use structure** | Headings, lists, code blocks — make information scannable |
| **Precision over formality** | "Server returned 503" is better than "The server encountered an issue" |
| **Assume good intent** | Tone is hard to convey in text; assume the best interpretation |

## Glossary

| Term | Definition |
|------|------------|
| Conveyance problem | The gap between what a sender means and what a receiver understands |
| Async communication | Communication that doesn't require simultaneous presence (writing, email) |
| Sync communication | Real-time communication (meetings, calls, in-person conversation) |
