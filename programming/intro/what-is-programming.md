# What Is Programming

## Description

Programming is the art and science of instructing a computer to perform tasks. This document covers what programming really means — beyond syntax — and sets the foundation for the rest of the Programming module.

## Content / Material

### Code Is a Means, Not an End

The goal of programming is not to write code — it's to **solve problems**. Code is a side effect of solving problems with computers.

Good programmers:
- Write code that is **readable** (humans read it more than machines execute it).
- Write code that is **maintainable** (it will change long after you write it).
- Write code that is **correct** (it does what it should) and **efficient** (it does it fast enough).

### Paradigms

Different problems call for different approaches:

| Paradigm | Core Idea | Languages |
|----------|-----------|-----------|
| **Imperative** | Step-by-step instructions | C, Python, Java |
| **Functional** | Pure functions, no side effects | Haskell, Elixir, Clojure |
| **Object-oriented** | Objects encapsulate data + behavior | Java, C++, Python, Ruby |
| **Declarative** | What to do, not how | SQL, HTML, Prolog |

Most modern languages support multiple paradigms.

### The Developer Toolbox

Beyond syntax, every developer needs:

- **Version control** — Git is non-negotiable.
- **Testing** — unit, integration, end-to-end.
- **Debugging** — reading stack traces, using a debugger, logging.
- **Refactoring** — improving code structure without changing behavior.
- **Code review** — giving and receiving constructive feedback.

### Programming Is About People

Code is written for two audiences: **machines** (that execute it) and **humans** (who read, maintain, and extend it). The best programmers optimize for the human side.

## Glossary

| Term | Definition |
|------|------------|
| Paradigm | A style or approach to programming |
| Side effect | Any state change that occurs outside a function's local scope |
| Refactoring | Restructuring code without changing its external behavior |
